### 🟦 Chunk 1 — Offload Heavy Work to Background Threads with Worklets 0.8 Bundle Mode

**Goal:** Move computationally expensive operations (data processing, animations, gesture handling) off the JS main thread using Worklets 0.8 Bundle Mode, resulting in a smoother, jank-free UI.

**Scope:**
- Upgrade `react-native-reanimated` and its `worklets` peer dependency to v0.8.0
- Identify one existing heavy computation in the app (e.g. complex data transformation on scroll, image filter, FFT)
- Refactor that computation into a `runOnUI` worklet using Bundle mode
- Benchmark frame rate before/after with Flashlist's `debugMode` or Perf Monitor
- Add a brief code comment explaining the threading model for future contributors

**Out of scope:**
- Full animation system rewrite
- Migrating all worklets to Bundle mode at once
- Shareable shared-memory primitives (a separate, more advanced chunk)

**Dependencies:**
- React Native 0.76+ (New Architecture strongly recommended)
- Existing `react-native-reanimated` setup

**Acceptance criteria:**
- [ ] App builds and runs without `worklets` version mismatch warnings
- [ ] Targeted computation executes on the UI thread without blocking JS
- [ ] Frame rate on the affected screen improves by at least 10% in the Perf Monitor
- [ ] A non-technical stakeholder can scroll or interact with the screen and confirm it feels smoother

**Estimated effort:** M

### 🟦 Chunk 2 — Add CSS Animations to SVG Components with Reanimated 4.3

**Goal:** Use Reanimated 4.3's new CSS animation support for SVG to ship polished, performant icon and illustration animations without writing custom worklets or keyframe logic.

**Scope:**
- Upgrade `react-native-reanimated` to v4.3.0
- Pick one static SVG component in the app (e.g. a loading spinner, success illustration, or chart)
- Convert it to an `Animated.Svg` component with a CSS `@keyframes`-style animation
- Verify the animation runs at 60 fps on both iOS and Android
- Document the pattern in the app's component storybook or README

**Out of scope:**
- Complex SVG path morphing animations
- Lottie replacement strategy
- Migrating all SVG components in one pass

**Dependencies:**
- Reanimated 4.3.0 installed
- SVG support via `react-native-svg` 14+

**Acceptance criteria:**
- [ ] SVG component animates smoothly on iOS and Android
- [ ] Animation is declared with CSS `@keyframes` syntax, not manual worklet interpolation
- [ ] Storybook or README documents the usage with a code snippet
- [ ] A non-technical stakeholder can identify the animated element and confirm it looks polished

**Estimated effort:** S

### 🟧 Chunk 3 — Migrate a Complex Native Feature to Expo (BLE + Background Audio Pattern)

**Goal:** Validate that Expo SDK 55 can handle demanding native requirements (persistent BLE connections, background audio) by replicating the Fieldy migration pattern, cutting app size and automating deployments.

**Scope:**
- Audit one complex native module in the project (BLE, background task, or media playback)
- Replace or wrap it with the equivalent Expo module (`expo-ble`, `expo-av`, or `expo-task-manager`)
- Configure EAS Update for OTA hotfixes on that module
- Measure app bundle size before/after (use `npx expo export --platform ios` + `du -sh`)
- Document the migration decision in `docs/adr/expo-native-modules.md`

**Out of scope:**
- Full migration of all native modules in one pass
- Publishing to App Store / Google Play (OTA update only)
- Bare RN → Expo managed workflow conversion

**Dependencies:**
- Expo SDK 55 (React Native 0.83)
- EAS account with OTA updates configured

**Acceptance criteria:**
- [ ] The target native feature works correctly via its Expo module equivalent
- [ ] App bundle size is equal or smaller than the baseline
- [ ] An OTA update can be pushed and applied on a test device without a full store build
- [ ] ADR is committed and reviewable by a non-technical stakeholder

**Estimated effort:** M

### 🟦 Chunk 4 — Migrate Navigator Config to React Navigation Static API

**Goal:** Adopt the React Navigation Static API to get automatic TypeScript type inference for routes and params, and zero-configuration deep linking, eliminating manual `ParamList` type declarations.

**Scope:**
- Upgrade `@react-navigation/core` to 7.17.x
- Convert one navigator (e.g. the main Tab or Stack navigator) from the dynamic API to `createStaticNavigation`
- Remove manual `RootParamList` type declarations for that navigator
- Configure deep link mappings using the static config's built-in `linking` support
- Verify navigation types are inferred correctly in at least 3 screens

**Out of scope:**
- Migrating all navigators in one pass (do one, document the pattern)
- React Navigation v8 alpha migration (wait for stable)
- Auth flow gating (separate chunk)

**Dependencies:**
- `@react-navigation/native` 7.x already installed
- TypeScript strict mode enabled

**Acceptance criteria:**
- [ ] Converted navigator produces no TypeScript errors on `useRoute()` params
- [ ] Deep link to at least one screen works on iOS and Android without additional config
- [ ] `npx tsc --noEmit` passes with zero new errors
- [ ] A non-technical stakeholder can test the deep link by pasting a URL into the simulator

**Estimated effort:** S

### ⚛️ Chunk 5 — Integrate Storybook MCP to Enable AI-Assisted Component Development

**Goal:** Connect Storybook 10.3's new MCP server to the team's AI coding assistant so it can write accurate stories, generate tests, and suggest fixes with full knowledge of the component library.

**Scope:**
- Upgrade Storybook to v10.3.0
- Enable the MCP server plugin in `.storybook/main.ts`
- Add the Storybook MCP endpoint to the AI assistant config (`.cursor/mcp.json` or Claude Desktop `mcp_config.json`)
- Validate with a real session: ask the agent to write a story for an existing component and fix one known UI bug
- Document the workflow in `docs/tooling/storybook-mcp.md`

**Out of scope:**
- Writing stories for the entire component library (the agent does that incrementally)
- Visual regression CI integration (separate chunk)
- React Native Storybook (this chunk targets the web Storybook used for design review)

**Dependencies:**
- Storybook already configured in the project
- An AI assistant that supports MCP (Cursor 0.43+, Claude Desktop, etc.)

**Acceptance criteria:**
- [ ] Storybook MCP server starts without errors alongside `storybook dev`
- [ ] AI assistant successfully generates a valid, renderable story for at least one component
- [ ] Agent identifies and proposes a fix for a known accessibility issue in the component
- [ ] A non-technical stakeholder can open Storybook and review the AI-generated story

**Estimated effort:** XS
