## 5-Day Action Plan

---

### 🟦 Chunk 1 — Implement Voltra Live Activities & Dynamic Island

**Goal:** Add native iOS Live Activities and Dynamic Island layouts to the app without writing a single line of Swift, using Voltra 1.0's JSX-first API.

**Scope:**
- Install `voltra` and configure its Expo Config Plugin
- Define a simple Live Activity layout using Voltra's SwiftUI primitives in JSX
- Connect the Live Activity to a real data source (e.g. order tracking, workout timer, delivery status)
- Test on a physical iOS device (Live Activities require hardware)

**Out of scope:** Android widgets (not supported by Voltra v1), custom SwiftUI animations, multi-island layouts.

**Dependencies:** Expo SDK ≥ 54, iOS 16.1+, physical iOS device for testing, Apple Developer account.

**Acceptance criteria:**
- A Live Activity appears on the Lock Screen and in the Dynamic Island during an active session
- Data updates in the Live Activity reflect app state in real time
- A non-technical stakeholder can trigger and see the Live Activity from within the app

**Estimated effort:** M

---

### 🟦 Chunk 2 — Upgrade to React Native 0.84 RC and Validate Hermes V1

**Goal:** Adopt RN 0.84 RC to benefit from Hermes V1 (now default), React 19.2.3, and new keyboard/accessibility events — and catch breaking changes before the stable release.

**Scope:**
- Bump `react-native` to `0.84.0-rc.0` in `package.json`
- Run the app on iOS and Android and resolve any deprecation warnings
- Confirm Hermes V1 is active via Metro/Hermes logs
- Test new `onKeyDown` / `onKeyUp` events on applicable text inputs and interactive components
- Document any migration notes in the project README

**Out of scope:** Production release with an RC version, full accessibility audit, custom Hermes build.

**Dependencies:** React Native ≥ 0.83 currently installed, Xcode ≥ 15, Android Studio Ladybug or newer.

**Acceptance criteria:**
- App builds and runs without errors on both platforms with RN 0.84 RC
- Hermes V1 is confirmed active in the build logs
- No new regressions detected in core user flows during smoke testing

**Estimated effort:** M

---

### 🟧 Chunk 3 — Prepare App Architecture for Expo SDK 55 Widgets & Live Activities

**Goal:** Lay the groundwork for the native Expo Widgets & Live Activities API (announced for SDK 55) so the team can adopt it on launch day without architectural debt.

**Scope:**
- Review Expo's announced roadmap and API surface for SDK 55 Widgets & Live Activities
- Identify 2–3 user-facing scenarios in the current app that would benefit from a widget or Live Activity
- Configure an App Group entitlement and create a shared `UserDefaults` data bridge readable by both the app and future widget targets
- Write a one-page design doc capturing the plan and store it in the team Notion

**Out of scope:** Actual widget implementation (pending SDK 55 stable release), Android widgets.

**Dependencies:** Expo SDK ≥ 54, Apple Developer account with App Groups entitlement enabled.

**Acceptance criteria:**
- App Group is configured and verified working (read/write from both app and a test target)
- Design doc is written, reviewed, and stored in team Notion
- A developer can follow the doc to implement the widget on the day SDK 55 ships

**Estimated effort:** S

### ⚛️ Chunk 4 — Integrate Vercel React Best Practices for AI Coding Agents

**Goal:** Adopt Vercel's curated React best-practice ruleset so Claude Code and other AI coding agents consistently generate idiomatic, production-quality React Native code in the project.

**Scope:**
- Download and review Vercel's [React best-practices markdown files](https://github.com/vercel-labs/agent-skills/tree/main/skills/react-best-practices/rules)
- Adapt and extend the rules to the project's stack (React Native + Expo Router)
- Add the adapted rules to `.cursor/rules` (Cursor), `.claude/skills` (Claude Code), or equivalent AI config
- Validate with 2–3 sample AI-generated components and verify output quality
- Commit the rules to the repository so all contributors benefit

**Out of scope:** Writing custom AI rules from scratch, integration with non-Claude/Cursor AI tools.

**Dependencies:** Claude Code, Cursor, or OpenCode installed locally, existing project codebase.

**Acceptance criteria:**
- AI-generated components follow the project's React Native conventions without manual correction
- Rules file is committed and its purpose documented in the README
- At least 2 AI-generated components are reviewed and approved by a developer without style corrections

**Estimated effort:** S

### 🟦 Chunk 5 — Add RNSEC Security Scanner to the CI/CD Pipeline

**Goal:** Automate detection of security vulnerabilities and hardcoded secrets in the React Native app by integrating the RNSEC CLI into every pull request check.

**Scope:**
- Install RNSEC CLI (`npx rnsec`) and run a baseline scan locally to produce an initial report
- Review and remediate all critical findings (hardcoded secrets, insecure network configs, etc.)
- Add a `rnsec` scan step to the GitHub Actions / EAS Build workflow
- Configure the CI step to block merges on critical severity findings

**Out of scope:** Full penetration testing, backend/API security audit, custom RNSEC rule development.

**Dependencies:** Existing CI pipeline (GitHub Actions or EAS Build), RNSEC CLI, Node.js ≥ 18.

**Acceptance criteria:**
- RNSEC scan runs automatically on every PR and the result is visible in the checks panel
- CI fails on critical findings and blocks merge until resolved
- The initial baseline report shows zero unresolved critical issues
- A non-technical stakeholder can read the scan summary in the CI dashboard

**Estimated effort:** S
