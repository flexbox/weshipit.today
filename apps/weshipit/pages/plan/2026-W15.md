### 🟦 Chunk 1 — Integrate On-Device AI Inference with RN ExecuTorch 0.8

**Goal:** Enable the app to run AI/ML models directly on-device (no server round-trip) by wiring ExecuTorch 0.8 to a Vision Camera frame processor and displaying real-time inference results.

**Scope:**

- Install and configure `react-native-executorch` v0.8.0 in a bare React Native project
- Set up Vision Camera v4 with a frame processor plugin
- Load a pre-trained on-device model (e.g. MobileNet for image classification)
- Display the classification label and confidence score in a UI overlay on the camera preview

**Out of scope:**

- Training or fine-tuning custom models
- Multi-model pipelines or model chaining
- Expo managed workflow (Vision Camera requires bare RN)
- NLP / Speech-to-Text hooks (separate task)

**Dependencies:**

- Bare React Native project (RN 0.83+ / SDK 55)
- Vision Camera v4 installed or added as part of this chunk
- Local model file prepared from ExecuTorch model hub

**Acceptance criteria:**

- App launches and camera preview renders without crash on iOS and Android
- Tapping "Analyze Frame" runs on-device inference and returns a visible classification label within 500 ms
- No network call is made during inference (verified via Reactotron network log)

**Estimated effort:** M

### 🟧 Chunk 2 — Set Up AI-Powered QA Agent with EAS Workflows

**Goal:** Automate visual UI verification for every pull request so the team gets screenshot-based regression reports without manual device testing.

**Scope:**

- Create an `eas-workflows.yml` that triggers on PR open/sync
- Install and configure `agent-device` to script navigation through at least 2 screens
- Add a GitHub comment step to post screenshot diff report to the PR
- Test the end-to-end flow on iOS Simulator in EAS infrastructure

**Out of scope:**

- Android CI configuration (deferred to a follow-up chunk)
- Full regression coverage across all screens
- Parallel multi-device runs
- Custom visual diffing thresholds

**Dependencies:**

- EAS account (free plan is sufficient)
- App built with Expo SDK 55+
- GitHub repository connected to EAS

**Acceptance criteria:**

- Opening a test PR triggers the EAS Workflow automatically within 2 minutes
- Agent-device navigates to Home and at least one detail screen without crash
- A GitHub comment with screenshot thumbnails appears on the PR before CI finishes
- Workflow duration is under 15 minutes on the EAS Free tier

**Estimated effort:** M

### 🟦 Chunk 3 — Configure Reactotron 3.9 MCP Server for AI-Assisted Debugging

**Goal:** Connect a coding AI assistant (Cursor, Claude, Copilot) directly to the running app via Reactotron's new MCP server, so developers can query Redux state and network logs without leaving the IDE.

**Scope:**

- Upgrade Reactotron to v3.9.0 (`reactotron-react-native`, `reactotron-redux`)
- Enable the built-in MCP server plugin in `ReactotronConfig.ts`
- Add the MCP endpoint URL to the IDE's MCP configuration (Cursor / VS Code settings)
- Write a short `DEBUGGING.md` guide documenting how to connect and sample queries

**Out of scope:**

- Custom Reactotron plugins or extensions
- Production monitoring or crash reporting
- Reactotron Cloud / remote debugging

**Dependencies:**

- Reactotron already integrated in the project (or added as part of this chunk)
- MCP-compatible IDE (Cursor ≥ 0.42 or VS Code with GitHub Copilot Agents)
- iOS Simulator or Android Emulator available locally

**Acceptance criteria:**

- Running the app in dev mode shows "MCP server listening on port 9090" in Reactotron
- The AI assistant responds to "show last 5 network requests" with accurate data from the running app
- Redux store root state is accessible via the AI agent without opening the Reactotron GUI
- `DEBUGGING.md` is merged to the repo and visible in GitHub

**Estimated effort:** S

### ⚛️ Chunk 4 — Migrate Router State to Fine-Grained Signal Subscriptions

**Goal:** Reduce unnecessary component re-renders triggered by monolithic router-state updates, improving perceived navigation performance on complex list or dashboard screens.

**Scope:**

- Audit current `router.state` subscriptions using React DevTools Profiler (record a before baseline)
- Replace broad subscriptions with scoped signal-based selectors using the TanStack Router signal graph API
- Re-run the Profiler to record an after baseline and document render count delta in the PR description
- Add one unit test per migrated selector to guard against regression

**Out of scope:**

- Migrating from React Navigation to TanStack Router (separate track)
- SSR or server-side routing changes
- Non-router state management (Redux, Zustand, etc.)

**Dependencies:**

- TanStack Router already in use in the project
- React DevTools Profiler available in the dev environment

**Acceptance criteria:**

- Profiler recording shows ≥ 30% fewer component renders during tab switches or route pushes
- All existing routes pass the existing navigation integration tests
- At least one signal selector is covered by a unit test

**Estimated effort:** S

### 🟦 Chunk 5 — Add Maestro E2E Baseline Suite with Preflight Deep-Link Injection

**Goal:** Ship a reproducible E2E baseline covering three critical user journeys (onboarding, authentication, main CTA), using Preflight's deep-link injection to isolate each screen and bypass multi-step setup flows.

**Scope:**

- Install `react-native-preflight` and Maestro CLI
- Configure Preflight's deep-link scheme in `app.json` (or `AndroidManifest` / `Info.plist`)
- Write three Maestro `.yaml` flows: (1) onboarding walkthrough, (2) login / logout, (3) main CTA action
- Capture and commit baseline screenshot snapshots for each flow to `e2e/snapshots/`

**Out of scope:**

- CI/CD pipeline integration (covered by Chunk 2)
- API mocking or network stubbing
- Performance benchmarking or load tests
- Android-specific deep-link edge cases (best-effort)

**Dependencies:**

- Maestro CLI installed locally (`brew install maestro` or equivalent)
- Deep-linking scheme already declared in the app manifest, or added as part of this chunk
- iOS Simulator or Android Emulator available

**Acceptance criteria:**

- `maestro test flows/` runs all 3 `.yaml` flows without error on a local iOS Simulator
- Preflight deep-links correctly bypass the navigation stack and land on the target screen within 2 seconds
- Three baseline snapshot files are committed to `e2e/snapshots/` and visible in the PR diff

**Estimated effort:** M
