### Working with branches

rename branch

```sh
git branch -m feature/old-branch feature/new-branch
```

fetch a remote branch

```sh
git checkout --track origin/feature/update-reviews
```

---

### Synchronize changes

```sh

```

```sh

```

```sh
git push [alias] [branch]
```

### Merge

```sh
git checkout develop
git pull --rebase
git merge --no-ff feature/branch

```

### [git-flow](http://nvie.com/posts/a-successful-git-branching-model/)

5 January 2010 â€” [Vincent Driessen](https://twitter.com/nvie)

- `main` is production with tags
- `develop` we work from here

<br>

- `feature` from `develop` and merge into `develop`
- `release` from `develop` and merge into `main` & `develop`
- `hotfix` from `main` and merge into `main` & `develop` / `release`

---

[git flow extension](https://github.com/nvie/gitflow)

```sh
git flow feature
git flow feature start <name> [<base>]
git flow feature finish <name>
```

### [github-flow](https://scottchacon.com/2011/08/31/github-flow/)

_31 August 2011 â€” [Scott Chacon](https://twitter.com/chacon)_

--

1. `main` is production

--

1. `feature` branch from `main`

--

1. often push on `origin`

--

1. open `pull-request` when you want

--

1. `merge` only after a pull-request review

--

1. **Deploy immediatly** after the merge on main
